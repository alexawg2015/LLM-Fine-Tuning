# Заняття 17: LLM Fine-Tuning in Production

## Бізнес-контекст

SaaS-компанія ShopFlow обробляє тисячі support emails на день. CTO просить автоматично перетворювати кожен email на структурований JSON для CRM:

```json
{
  "sender": "Marcus Webb",
  "product": "Payments",
  "category": "technical",
  "urgency": "critical",
  "summary": "Payment processing down since 2 AM on Black Friday, $4,000 in lost sales."
}
```

Це дозволяє:
- Routing **critical** incidents у PagerDuty для негайної ескалації
- Виявлення billing-патернів що ведуть до churn
- Скорочення time-to-first-response

**Проблема:** За умовою задачі Llama 3.3 70B з промптом дає urgency accuracy ~60% — недостатньо для автоматичної ескалації¹. Плюс inference 70B при великому трафіку коштує сотні доларів на місяць.

**Гіпотеза:** Fine-tuned Llama 3.1 8B перевершить base модель і конкуруватиме з Llama 3.3 70B на цій задачі при значно дешевшому inference.

> ¹ У нашому експерименті Llama 3.3 70B через OpenRouter API з оптимізованим промптом показала 80% urgency accuracy — вище ніж заявлено в умові. Розбіжність пояснюється різницею в промптах і синтетичним характером нашого eval set. Тим не менше fine-tuned 8B демонструє конкурентні результати при значно нижчій вартості.

## Платформа

Завдання виконувалось на **Kaggle Notebooks** (T4 GPU) замість Google Colab — через блокування Colab акаунту під час виконання. Стек залишився ідентичним умові: Unsloth + QLoRA + transformers.

## Методологія

### Крок 1 — Eval Set (30 прикладів)
Створено вручну **до** будь-якого тренування щоб зафіксувати чесну "лінійку" для всіх наступних вимірювань.

| Тип | Кількість |
|---|---|
| Critical urgency (ескалація PagerDuty) | 7 |
| Anonymous sender | 3 |
| Sarcasm / passive-aggressive | 3 |
| Multiple issues в одному email | 4 |
| Нормальні чіткі кейси | 13 |

### Крок 2 — Training Set (300 прикладів)
Синтетично згенеровані приклади для e-commerce платформи ShopFlow. Перевірено що жоден приклад з training set не присутній в eval set — виключає можливість що модель бачила правильні відповіді під час тренування.

| Блок | Кількість | Зміст |
|---|---|---|
| Звичайні кейси | 150 | Всі продукти і категорії |
| Складні urgency | 60 | Ввічливий тон але критична ситуація |
| Anonymous sender | 20 | Edge case без підпису |
| Multiple issues | 20 | Кілька проблем в одному email |
| Sarcasm | 20 | Пасивно-агресивний тон |
| Різноманітні | 30 | Додаткове покриття |

### Крок 3 — Baseline Evaluation
Llama 3.1 8B (без fine-tuning) на eval set з фіксованим промптом і `do_sample=False` (greedy decoding) для відтворюваності результатів.

### Крок 4 — Порівняння з Llama 3.3 70B
Llama 3.3 70B через OpenRouter API з **ідентичним промптом** і `temperature=0`. Це закриває основну гіпотезу курсу.

### Крок 5 — Fine-Tuning
QLoRA 4-bit, LoRA r=16, alpha=32, 3 epochs, LR=2e-4, batch=2, grad_accum=8.
Час тренування: **11 хвилин** на T4 GPU. Вартість: **$0** (Kaggle free tier).

### Крок 6 — Re-Evaluation
Той самий eval set, той самий промпт — для чистого порівняння з baseline.

### Збереження адаптера
LoRA адаптер збережено окремо від базової моделі — лише навчені ваги, не вся модель (8B параметрів).

Фактичний розмір адаптера: **176 MB** (умова завдання очікувала ~50 MB). Різниця пояснюється конфігурацією:

| Параметр | Умова | Наш експеримент |
|---|---|---|
| LoRA rank (r) | не вказано | 16 |
| Target modules | не вказано | 7 (q, k, v, o, gate, up, down proj) |
| Layers | 32 | 32 |
| LoRA ваги | ~50 MB | ~140 MB |
| Токенайзер | не включено | ~17 MB |
| Конфіги | — | ~5 MB |
| **Разом** | **~50 MB** | **~176 MB** |

~50 MB відповідає меншій конфігурації (r=8, лише attention layers без MLP). Наша конфігурація r=16 з повним набором target_modules дає кращу якість але більший адаптер. Це свідомий trade-off.

Ваги зберігаються на Kaggle (не в репо — GitHub має ліміт 100 MB на файл).

## Результати

### Основна таблиця порівняння

| Метрика | Base 8B | Fine-tuned 8B | Δ | Llama 3.3 70B |
|---|---|---|---|---|
| json_valid | 100.0% | 100.0% | → | 100.0% |
| exact_match | 30.0% | **56.7%** | ↑ +26.7% | 50.0% |
| sender accuracy | 90.0% | **100.0%** | ↑ +10.0% | 100.0% |
| product accuracy | 63.3% | **76.7%** | ↑ +13.4% | 73.3% |
| category accuracy | 80.0% | **93.3%** | ↑ +13.3% | 73.3% |
| urgency accuracy | 60.0% | 76.7% | ↑ +16.7% | **80.0%** |
| summary present | 100.0% | 100.0% | → | 100.0% |
| avg tokens | 145.8 | **56.1** | ↓ -61.5% | 53.3 |
| avg latency | ~1.1 сек | ~1.1 сек | → | 2.39 сек |

### Urgency детально

| Рівень | Base 8B | Fine-tuned 8B | Llama 3.3 70B |
|---|---|---|---|
| **critical** (7) | ~43% | **6/7 = 86%** | 3/7 = 43% |
| high (9) | ~60% | **8/9 = 89%** | 9/9 = 100% |
| medium (4) | ~25% | 1/4 = 25% | 3/4 = 75% |
| low (10) | ~80% | 8/10 = 80% | 9/10 = 90% |

**Ключовий результат:** fine-tuned 8B поступається 70B по загальній urgency accuracy (76.7% vs 80.0%), однак для нашої задачі найважливішим є **critical** urgency — саме він визначає ескалацію в PagerDuty. Тут fine-tuned 8B значно перевершує 70B (86% vs 43%). Це означає що 70B пропустить більше половини реальних критичних інцидентів.

### Висновок по гіпотезі
Fine-tuned 8B **перевершує 70B** по 5 з 7 метрик (exact_match, category, product, critical urgency, latency) при вдвічі нижчій вартості inference. Гіпотеза **підтверджена** для практичної бізнес-задачі.

## Аналіз результатів

### Що вийшло і чому

**Exact match +26.7% (30% → 56.7%)** — найбільший абсолютний приріст. Base модель правильно генерувала формат JSON (100% json_valid) але помилялась в 1-2 полях одночасно. Fine-tuning навчив узгодженим патернам: payment failure → technical + critical, feature request → low urgency.

**Category accuracy +13.3% (80% → 93.3%)** — fine-tuned 8B перевершив навіть 70B (93.3% vs 73.3%). Конкретні приклади для кожної категорії допомогли засвоїти межі між billing/technical/account на реальних e-commerce сценаріях.

**Critical urgency 43% → 86%** — найважливіший для бізнесу результат. Base модель оцінювала тон листа (ввічливий = не критично), а не бізнес-наслідки. Fine-tuning навчив: "магазин не приймає платежі просто зараз" = critical, незалежно від тону.

**Avg tokens -61.5% (145 → 56)** — base модель продовжувала генерувати текст після JSON. Fine-tuning навчив зупинятись одразу після `}`. Прямий вплив на inference cost і latency.

### Що не вийшло

**Medium urgency — слабке місце (25%)** — з 4 medium прикладів модель правильно визначила лише 1. Причини: мала кількість medium прикладів у training set відносно їх суб'єктивності, розмита межа між medium і high, і лише 4 приклади в eval set роблять цю метрику статистично ненадійною.

**Class imbalance в training set** — technical домінує (176/300 = 59%). При малому датасеті це схиляє модель до частішого вибору технічних категорій на граничних випадках.

**Train/serve skew** — множинні конфлікти версій бібліотек (transformers 5.10.2 vs Unsloth, trl 1.5.1 vs unsloth-zoo) призвели до кількох рестартів сесії. Вирішення: Docker-контейнер з фіксованими версіями (Заняття 13).

## Додаткове дослідження: Validation Set (Експеримент 2)

**Мета:** перевірити чи допоможе train/val split (80/20), менший LR (1e-4) і 5 epochs покращити urgency accuracy.

| Метрика | Base 8B | FT v1 | FT v2 | Δ v1→v2 |
|---|---|---|---|---|
| exact_match | 30.0% | 56.7% | 53.3% | ↓ -3.4% |
| urgency accuracy | 60.0% | 76.7% | 73.3% | ↓ -3.4% |
| medium urgency | ~25% | 1/4 = 25% | 2/4 = 50% | ↑ +25% |
| high urgency | ~60% | 8/9 = 89% | 6/9 = 67% | ↓ -22% |

**Висновок:** medium покращився (25% → 50%) але high погіршився (89% → 67%) — класичний trade-off при малому датасеті. Коли відкладаємо 60 прикладів на validation, модель навчається на 240 замість 300, і кожен приклад стає критично важливим. Validation set корисний при 1000+ прикладах де статистична надійність вища.

## Бізнес-рекомендація

Fine-tuned Llama 3.1 8B є кращим вибором для production ніж Llama 3.3 70B: він розпізнає critical інциденти вдвічі краще (86% vs 43%), працює вдвічі швидше ( ~1.1 сек vs 2.39 сек), і коштує значно менше ( ~$250/міс vs ~$650/міс при 50K emails/день).

**Головний напрям покращення** — збільшення training set. Генерація 1,000-2,000 прикладів не потребує витрат, а при накопиченні реальних листів можна використати активне навчання: AI відбирає найбільш корисні приклади з рівномірним розподілом по urgency і категоріям. Це реалістично дозволить досягти 90%+ urgency accuracy і повністю автоматизувати routing.

Якщо збільшення датасету неможливе в короткостроковій перспективі — рекомендується гібридний підхід: автоматичний routing для `critical` (86% accuracy) і `low` (80%), з людською верифікацією для `medium` де обидві моделі показують слабкі результати.

## Cost & Breakeven

Розрахунок: ~350 input + 56 output tokens на email, ціни Together AI ($0.88/1M для 70B, $0.18/1M для 8B API). Вартість GPU хостингу — фіксована незалежно від трафіку.

| Варіант | 10K emails/день | 50K emails/день | Latency |
|---|---|---|---|
| Llama 3.3 70B (Together AI, $0.88/1M) | ~$130/міс | ~$650/міс | 2.39 сек |
| Llama 3.1 8B API (Together AI, $0.18/1M) | ~$26/міс | ~$130/міс | ~1 сек |
| Fine-tuned 8B (Hyperstack A4000 16GB, $0.15/год) | ~$108/міс | ~$108/міс | ~1.1 сек |
| Fine-tuned 8B (vast.ai RTX 3090 24GB, $0.20/год) | ~$145/міс | ~$145/міс | ~1.1 сек |
| Fine-tuned 8B (vast.ai RTX 4090 24GB, $0.53/год) | ~$380/міс | ~$380/міс | ~1.1 сек |
| Fine-tuned 8B (AWS g4dn.xlarge, $0.526/год) | ~$380/міс | ~$380/міс | ~1.1 сек |
| **Training cost** | **$0** (Kaggle free T4) | **$0** | — |

**Ключовий висновок:** порівнюємо fine-tuned 8B (наш результат, кращі метрики) з Llama 3.3 70B API (найближча альтернатива по якості):


- До ~10K emails/день → **70B API** ( ~$130/міс) дешевша за будь-який GPU хостинг
- Від ~10K emails/день → **Fine-tuned 8B на Hyperstack A4000** ($108/міс) дешевша за 70B API
- При 50K emails/день → економія **~$540/місяць** ($108 vs $650)

8B API наведена для довідки — вона дешевша але поступається по якості (urgency 60% vs 76.7%).
